D- The Requirements
The BMI App. The objective of this project is to build an Android app that computes the BMI (Body
Mass Index) given the weight w and height h of a person. If w is measured in kilograms and h in
meters, then the index can be computed using the formula: BMI = w/h2. The architecture of the
app must adhere to the MVC design pattern (see the Design section for details).
The UI & Behaviour. As shown in Fig. D1.1, the UI
(user interface) of the sought app consists of six
vertically aligned elements (aka widgets):
1) A label that says "Your Weight".
2) A textbox for entering the weight.
3) A label that says "Your Height".
4) A textbox for entering the height.
5) A button with caption "COMPUTE BMI".
6) A label for displaying the answer.
Each of the six widgets must extend across the
entire width of the device
(phone or tablet)
regardless of device dimensions and whether held
in portrait or landscape. Text must be left aligned
except for the button caption and the computed
index which must be centred.
In terms of behaviour, the app must display the
computed index whenever the button is tapped.
The index (which is a real number) should be
rounded to two decimal, and must appear in large
blue text.
It should be noted that no requirement specification would be complete if it did not clearly
define the validity condition of the input and how the app is to behave should the input be
invalid. Failing that, the app will have bugs and vulnerabilities that can be exploited. For our app,
we consider the input to be valid if w and h are positive real numbers. And in this first release,
we will assume this validity as a precondition. Recall “Design By Contract”. This means the app
should not attempt to validate the entries. If the user made a non-numeric entry, or a non-
positive one, then it is the user who bears the responsibility for the consequences (a wrong
answer or the app crashing or any other outcomes).

The Design
The MVC Architecture. We will adopt the Model-View-Controller (MVC) design pattern for
structuring our app. In MVC, an app must have three components: the model (an abstract
component that encapsulates the data and the computation), the view (a component
responsible for interacting with the user), and finally the controller (a component that liaises
between M and V, thereby embodying the app's behaviour).
In Android lingo:
• The model is represented by a POJO (Plain-Old Java Object) that is Android agnostic.
As such, it can be tested on its own and can be ported as-is to other platforms such as a
desktop application or a web service. We will call our model "BMIModel.java".
• The view is known as a layout and is represented by an XML document, a file whose
name ends with an xml extension.
• The controller is known as an activity. It is represented by a Java class1 (like the model)
so its name has a java extension.
The Model's API. The model class is responsible for performing the BMI computation. We will
design it using the functional paradigm (Section L0.2.a), and as such, we should have a method2
in it like this:
public static double getBMI(double weight, double height)
We used double3 for the two parameters and the method return because these quantities are
all real numbers. For example, a weight of 90.0 kg and a height of 1.8 m lead to a return of
27.77777 (use your calculator to check).
The plan is thus to have the controller grab the weight and height from the view; pass them
to the getBMI method in the model; and then send the return value back to the view.
But the view does not deal with typed data: to the view, everything is a sequence of characters4;
i.e. strings, so there are no integers or reals. Hence, our model must also have a method
to convert a string like "1.8" to the double like 1.8. It must also have a method to convert
and round the BMI from a double like 27.77777 to a string like "27.8".

The Deployment
Integration Testing. A typical app contains several components: one or more view, one or more
activity, and one or more model class. We learned how to test each component by itself (this is
unit testing) but even if all components pass their unit tests, the overall app may not work. This
typically happens because components do not "talk" properly with each other. For example, a
component that expects a positive integer as a precondition will pass its own unit test but will
crash if chained with a component that feeds it a negative integer.
Integration testing involves testing the app as a whole after unit testing all its components.
We do this for an Android app by deploying (running) the app as a whole and testing various
scenarios in an end-to-end fashion. You test the view by ensuring it meets the requirement
regardless of the device dimension or how it is held (portrait / landscape). Next enter several
valid inputs (including boundary cases at the edges of the allowed range) and verify that the
outputs meet the requirement, not only in terms of value but also in formatting (e.g., rounding,
text size, colour, etc.).
Design By Contract. Note that you only test valid inputs; i.e., ones that meet the pre-condition.
If you supply an invalid input such as zero for the height or a non-numeric entry for either the
weight or the height, then we, the developers, are not responsible for whatever happens. This is
the essence of input validity as a precondition versus a requirement. It assigns responsibility on
the user (or client) of a piece of software rather than its implementer. Put differently, the app
is supposed to behave as required if, and only if, the pre-condition is met. Had the designer
wanted to allow such invalid inputs then the requirement would have been relaxed, and we
would have validated the input and issued error messages.
Deployment Options. As outlined in Chapter 0, you deploy the app from the Run menu of the
IDE. To do that, you first select a device (from the Run/Select menu) and then run. You can
deploy on a virtual device (one that emulates a real device) or on a physical one. It is a good idea
to have a couple of virtual devices of different sizes (e.g. a phone and a tablet) so you can test
the view on them. To deploy on a physical device, make sure it has USB Debugging enabled and
then connect it to your computer. The app will be side-loaded and installed on the device. Note
that your app will remain on the device even after you disconnect it from your machine. Verify
this by going to the apps screen on the device and locate your app by looking for its name (BMI).
It will have a default green Android badge.
Note: if you prefer to deploy your app from the command line rather than from the IDE, go
To the project's directory and issue the command:
